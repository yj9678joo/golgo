# Golgo 에이전트 작업 지침

## 프로젝트 개요

Golgo(고르고)는 증권사 Open API와 LLM을 결합해 개인 맞춤형 종목 분석 및 리밸런싱 가이드를 제공하는 서비스다.

핵심 목표:

- 한국투자증권/미래에셋 Open API로 보유 종목, 잔고, 비중을 동기화한다.
- 사용자의 투자 성향과 전략을 바탕으로 개인화된 리밸런싱 가이드를 제공한다.
- Gemini/GPT/Claude 기반 LLM으로 7단계 심층 투자 분석 리포트를 생성한다.
- 목표 비중과 현재 비중의 차이를 계산해 매수/매도 필요 수량을 산출한다.
- 주식 초보자도 이해할 수 있는 직관적인 대시보드와 리스크 체크리스트를 제공한다.
- 초기에는 수동 매매 가이드를 제공하고, 이후 자동 리밸런싱으로 확장 가능한 구조를 유지한다.

## 상세 개발 규약 참조

작업 영역별 상세 규약은 아래 파일을 우선 확인하고 따른다.

- 프론트엔드 작업: `.codex/front-end/font_end.md`
- 백엔드 작업: `.codex/back-end/back_end.md`

프론트엔드 또는 백엔드 작업을 시작하기 전에 해당 규약 파일을 먼저 읽고, 기술 스택과 디렉토리 구조, 테스트 기준을 확인한다.

## 공통 작업 지침

LLM의 일반적인 코딩 실수를 줄이기 위한 행동 지침이다. 프로젝트별 지침과 병합하여 사용한다.
본 지침은 속도보다 신중함에 우선순위를 둔다. 단, 사소한 작업은 상황에 맞게 판단한다.

### 구현 전 사고

- 가정하지 않는다. 모호함을 숨기지 않는다. 트레이드오프를 명확히 밝힌다.
- 구현을 시작하기 전 자신의 가정을 명시적으로 기술한다. 불확실한 경우 질문한다.
- 해석의 여지가 여러 가지라면 임의로 선택하지 말고 대안들을 제시한다.
- 더 간단한 접근 방식이 있다면 제안한다. 정당한 사유가 있다면 사용자의 요청에 반대 의견을 제시한다.
- 불분명한 부분이 있다면 작업을 중단한다. 혼란스러운 부분을 구체적으로 언급하며 질문한다.

### 단순성 우선

- 문제를 해결하는 최소한의 코드만 작성한다. 추측에 기반한 코드는 배제한다.
- 요청되지 않은 기능은 추가하지 않는다.
- 일회성 코드를 위해 추상화 계층을 만들지 않는다.
- 요청되지 않은 유연성이나 설정 가능성을 고려하지 않는다.
- 발생 불가능한 시나리오에 대한 예외 처리를 하지 않는다.
- 200줄의 코드를 50줄로 줄일 수 있다면 코드를 다시 작성한다.
- 시니어 엔지니어가 보기에 지나치게 복잡한 코드라면 단순화한다.

### 정밀한 수정

- 필요한 부분만 수정한다. 본인이 만든 코드의 뒷정리만 수행한다.
- 인접한 코드, 주석, 포맷을 임의로 개선하지 않는다.
- 망가지지 않은 부분을 리팩토링하지 않는다.
- 본인의 스타일과 다르더라도 기존 스타일을 따른다.
- 작업과 무관한 데드 코드를 발견하면 보고하되 직접 삭제하지 않는다.
- 본인의 수정으로 인해 불필요해진 임포트, 변수, 함수는 제거한다.
- 기존에 존재하던 데드 코드는 요청이 없는 한 그대로 둔다.
- 변경된 모든 라인은 사용자의 요청사항과 직접적으로 연결되어야 한다.

### 목표 중심 실행

- 성공 기준을 정의하고 검증될 때까지 반복한다.
- 작업을 검증 가능한 목표로 변환한다.
  - 유효성 검사 추가: 잘못된 입력에 대한 테스트 작성 후 통과 확인
  - 버그 수정: 버그를 재현하는 테스트 작성 후 통과 확인
  - 리팩토링: 리팩토링 전후의 테스트 통과 확인
- 다단계 작업은 간략한 계획을 수립한다.
  1. 단계: 수행할 작업을 정의한다. 검증: 확인 사항을 명시한다.
  2. 단계: 수행할 작업을 정의한다. 검증: 확인 사항을 명시한다.
  3. 단계: 수행할 작업을 정의한다. 검증: 확인 사항을 명시한다.
- 성공 기준이 명확해야 독립적인 작업이 가능하다. "작동하게 만들기"와 같은 모호한 기준은 피한다.

## AI 분석 프레임워크

LLM 분석 기능은 50년 경력의 베테랑 분석가 페르소나를 기준으로 한다.

분석 리포트는 다음 7단계를 중심으로 구성한다.

1. 기업 개요 및 수익 모델
2. 산업 및 경쟁 구조
3. 재무 분석
4. 밸류에이션 분석
5. 실적 및 컨콜 분석
6. 거시경제 및 정책 영향
7. 촉매와 리스크

밸류에이션 분석은 PER, PEG, PBR, PSR을 교차 검증해야 한다. 지표가 서로 모순될 때는 거시경제 환경과 연결해 경고 문구와 비중 축소 가능성을 함께 제시한다.

상세 페르소나 로직은 `프로젝트_개요.md`의 'AI 분석 프레임워크 (7단계 페르소나 로직)'을 참고한다.

## 시스템 워크플로우

기능 구현 시 아래 흐름을 기준으로 설계한다.

1. 증권사 API로 잔고 및 종목 데이터를 가져온다.
2. 사용자 성향, 현재 포트폴리오, 시장 데이터를 조합해 프롬프트를 구성한다.
3. LLM이 전문 분석가 페르소나로 분석을 수행한다.
4. LLM 응답을 엄격한 JSON 구조로 파싱해 대시보드 UI에 매핑한다.
5. 사용자는 리포트를 확인한 뒤 직접 매매한다.
6. 자동 주문 기능은 후속 확장으로 분리한다.

## 구현 원칙

- 요청된 범위 안에서 필요한 최소 변경만 수행한다.
- 추측으로 기능을 추가하지 않는다.
- 기존 코드 스타일과 디렉토리 구조를 우선 따른다.
- 프론트엔드/백엔드 상세 규약과 충돌하는 구현을 하지 않는다.
- API는 엔드포인트, 요청 JSON, 응답 JSON 예시를 함께 고려한다.
- LLM 응답은 자유 텍스트가 아니라 구조화된 출력으로 제한한다.
- 증권사 API Key, Access Token, 계좌 관련 민감 정보는 암호화하고 로그에 남기지 않는다.
- 증권사 API Rate Limit을 고려해 캐싱, 큐, 배치 처리를 설계한다.

## 검증 기준

- 프론트엔드 변경은 `.codex/front-end/font_end.md`의 테스트 및 빌드 기준을 따른다.
- 백엔드 변경은 `.codex/back-end/back_end.md`의 테스트 및 API 규약을 따른다.
- 실제 브라우저 검증이 필요한 경우에는 `browser-test` 스킬을 명시적으로 요청한다.
- 문서만 변경한 경우에는 변경 파일을 다시 읽어 내용 반영 여부를 확인한다.
- 테스트나 린트를 실행하지 못한 경우 그 이유를 결과에 명시한다.

## 커밋 지침

- 변경 사항에는 본 지침을 엄격히 적용하여 커밋을 수행한다.
- 커밋 전 변경된 파일 목록과 핵심 diff 요약을 간략히 보고한다.
- 변경된 사항이 프로젝트 테스트 및 린트 검사를 통과했는지 확인하고 결과를 알린다.
- 검증이 완료되면 메시지와 함께 변경 사항을 커밋한다.
- 사용자가 명시적으로 요청하기 전에는 push하지 않는다.
- push가 완료되면 커밋 SHA, 브랜치 이름, 성공 결과를 짧게 요약한다.
- push 성공 후 해당 commit SHA로 실행된 GitHub Actions를 자동 조회한다.
- workflow 등록을 최대 60초간 대기한 뒤 완료될 때까지 감시한다.
- GitHub Actions의 성공 여부와 실패한 job/check를 보고한다.
